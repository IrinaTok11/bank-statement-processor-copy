Methodology — Bank Statement Processor

How the automation reads statements, standardizes formats, categorizes transactions, screens for anomalies, and generates forensic findings

Repository Back to portfolio

Architecture — overview

  • Multi-file input: All .xlsx/.xls files in run/ workspace
  • Deterministic pipeline: Format detection → standardization → categorization → forensic screening → five-sheet workbook
  • Full traceability: Every transaction includes Source_File, Source_Sheet, Source_Row for audit verification
  • Explainable rules: All detection thresholds and categorization logic documented in Config & Thresholds sheet

Full architecture (GitHub)

Data flow

  1. Detect inputs. Scan run/ for all .xlsx/.xls files (ignores Excel temp files like ~$...)
  2. Identify format. Extract bank name from filename (first word before underscore), auto-detect header row, determine column layout (Date / Description / Debit / Credit / Balance)
  3. Load statements. Read each file with format-specific parsers (Chase, Santander, ABSA configs + generic fallback)
  4. Standardize schema. Convert all transactions to canonical fields: Date, Description, Debit, Credit, Bank, Source_File, Source_Sheet, Source_Row
  5. Normalize text. Clean descriptions (remove transaction IDs, normalize whitespace, extract vendor names)
  6. Categorize transactions. Apply keyword-based rules to assign Operation_Type (Vendor Payment, Customer Receipt, Payroll, Tax, Banking Fee, etc.) and Direction (Inflow/Outflow)
  7. Apply forensic screening. Run six detection algorithms:
    • Duplicate Payment Detection (same vendor ±1% amount within 3 days, min $100 debit)
    • Split Payment Patterns (2+ same-day payments to same vendor, total >$10k, any single >$5k)
    • Vendor Concentration Growth (baseline→latest month spend >2x, baseline >$1k)
    • Round Amount Clustering (exact $1,000 increments)
    • Weekend Activity Alerts (large debits on Sat/Sun, excludes taxes/payroll/recurring, threshold = p95 debit × 1.25)
    • New Counterparty Flags (first-time vendors with debits >$50k)
  8. Generate findings report. Group related transactions into alerts, assign priority (P1/P2/P3), create executive summary and detail table
  9. Build analytics. Calculate cash metrics (period dates, transaction counts, opening/ending balances, net change), monthly trends, Benford's Law distribution, vendor concentration analysis, category breakdowns
  10. Document methodology. Export complete rule definitions, thresholds, rationale, and run-specific parameters to Config & Thresholds sheet
  11. Perform QA validation. Compare expected vs. actual values for key metrics (monthly totals, vendor spending, category sums) with tolerance checks
  12. Write workbook. Export five sheets: Transactions, Findings & Alerts, Evidence Index, Analytics Dashboard, Config & Thresholds

Format detection strategy

Bank identification

Extract bank name from filename: CHASE_20230515.xlsx → bank = "CHASE"

Important: The processor uses the first word before underscore in the filename as the bank identifier.

  • NORTHBRIDGE_May_June_2023.xlsx → Bank = "NORTHBRIDGE"
  • GLOBAL_BUSINESS_BANK_USD.xlsx → Bank = "GLOBAL"
  • SANTANDER_Statement.xlsx → Bank = "SANTANDER"

Header auto-detection

Search rows 1-50 for header signatures:

  • Look for rows containing: Date AND Description AND (amount columns)
  • Amount columns: Debit/Credit OR In/Out OR Amount OR Money In/Money Out
  • First matching row becomes header row index

Format type classification

Two layout types supported:

  • Debit/Credit separate — two columns (Chase, Santander, ABSA, most banks)
  • Amount signed — single column with +/− values (less common, handled by generic parser)

Column mapping

Case-insensitive, space-tolerant keyword matching:

  • Date → Date / Transaction Date / Trans Date / Posting Date / VALUE DATE
  • Description → Description / Details / Transaction / Memo / Narrative
  • Debit → Debit / Out / Money Out / Withdrawal / Payment / Payments Out
  • Credit → Credit / In / Money In / Deposit / Receipt / Payments In
  • Balance → Balance / Running Balance / Closing Balance
  • Transaction ID → ID / Reference / Transaction ID / Ref / Check #

Direction classification

  • Inflow — Credit amount present, Debit empty/zero
  • Outflow — Debit amount present, Credit empty/zero
  • Mixed — Both Debit and Credit present (rare, flagged for review)

Vendor normalization

Extract clean vendor name from description:

  • Remove transaction IDs (patterns like #12345, INV 6789)
  • Remove date stamps (2023-05-15, 05/15/23)
  • Strip common prefixes (ACH PAYMENT -, MERCHANT SETTLEMENT -)
  • Normalize whitespace (multiple spaces → single space)
  • Uppercase for matching consistency

Forensic screening rules

1. Duplicate Payment Detection

Rule: Same vendor within ±1% amount AND within 3 days (min debit $100)

Rationale: Catches invoice rounding variations (e.g., $1,084.59 vs. $1,084.60) while filtering noise from small transactions

Priority: P1 (High) — potential duplicate invoicing or payment processing error

Implementation:

  • Group transactions by Vendor_Normalized
  • For each vendor, compare all pairs of debits ≥$100
  • Flag if: |amount1 - amount2| / amount1 ≤ 0.01 AND |date1 - date2| ≤ 3 days

2. Split Payment Patterns

Rule: 2+ payments to same vendor on same day; total >$10,000 (flag if any single >$5,000)

Rationale: Detects approval limit circumvention (e.g., splitting $12k invoice into 2×$6k payments to stay under $10k authorization threshold)

Priority: P2 (Medium) — potential policy circumvention or fraud scheme

Implementation:

  • Group by Vendor_Normalized + Date
  • Count transactions per group
  • Flag if: count ≥ 2 AND sum(debits) > $10,000 AND any single debit > $5,000

3. Vendor Concentration Growth

Rule: Baseline (first active month) → latest month spend >2x; baseline >$1,000

Rationale: Identifies rapidly increasing vendor dependency, potential kickback arrangements, or procurement policy failures

Priority: P2 (Medium) — concentration risk, vendor relationship investigation needed

Implementation:

  • For each vendor, calculate monthly spending
  • Baseline = first month with spending ≥$1,000
  • Latest = most recent month in dataset
  • Flag if: latest_spend / baseline_spend > 2.0

4. Round Amount Clustering

Rule: Exact $1,000 increments (e.g., $5,000.00, $10,000.00, $25,000.00)

Rationale: Round amounts suggest estimates, manual entries, or potential fabrication (real invoices typically have cents)

Priority: P3 (Low) — review for legitimacy, cross-reference with supporting documentation

Implementation:

  • Check if: amount % 1000 == 0 AND amount ≥ $1,000
  • Exclude categories: Payroll (often round), Tax (often round), Transfers (often round)

5. Weekend Activity Alerts

Rule: Dynamic threshold (p95 debit × 1.25) for large debits on Sat/Sun

Rationale: Unusual business-hours activity may indicate unauthorized access or emergency disbursements requiring review

Priority: P3 (Low) — contextual review (may be legitimate)

Implementation:

  • Calculate p95 = 95th percentile of all debits
  • Threshold = p95 × 1.25
  • Flag weekend debits if: weekday in [Saturday, Sunday] AND debit > threshold
  • Exclude categories: Tax, Payroll, Recurring (legitimate weekend processing)

6. New Counterparty Flags

Rule: First-time vendors with debits >$50,000

Rationale: Large initial transactions require enhanced due diligence (vendor verification, contract review, authorization checks)

Priority: P2 (Medium) — due diligence investigation

Implementation:

  • Sort transactions by date (ascending)
  • For each vendor, identify first transaction
  • Flag if: first transaction is a debit AND amount > $50,000

Benford's Law analysis

Purpose

Statistical anomaly detection for fabricated transactions. Legitimate financial data follows Benford's distribution (first digit frequency: 1=30.1%, 2=17.6%, 3=12.5%, etc.). Fabricated data typically shows flat distribution.

Implementation

  • Extract first digit from all debits (e.g., $5,432.10 → 5)
  • Calculate actual frequency distribution
  • Compare to expected Benford's distribution
  • Flag significant deviations (>5 percentage points)

Output

Table in Analytics Dashboard showing:

  • Digit (1-9)
  • Expected % (Benford's Law)
  • Actual % (observed data)
  • Deviation (percentage points)

Limitations

Small datasets (<200 transactions) may not follow Benford's Law. Use as screening tool, not definitive proof.

Outputs

Sheet 1: Transactions

Standardized table with columns:

  • Date, Description, Debit, Credit, Bank
  • Source_File, Source_Sheet, Source_Row — audit traceability
  • Description_clean, Vendor_Normalized — text processing results
  • Operation_Type, Direction — categorization outputs
  • Month, Weekday — temporal analysis fields

Sheet 2: Findings & Alerts

Executive Summary: Total findings, priority breakdown (P1/P2/P3)

Detail Table: Columns include Alert_ID, Priority, Category, Vendor, Details, Related_Txns, Total_Amount, First_Date, Last_Date, Review_Status

Sheet 3: Evidence Index

Transaction-level drill-down: One row per transaction involved in an alert

Columns: Alert_ID, Category, Vendor, Group, Txn_ID, Txn_Date, Debit, Credit, Bank, Source (formatted as "file | sheet | row")

Sheet 4: Analytics Dashboard

  • Cash flow metrics: Analysis period (dates), total transactions, opening/ending balances, net cash change
  • Monthly trends: Inflows, outflows, net by month
  • Benford's Law: First-digit distribution analysis
  • Vendor concentration: Top vendors by spending with % of total
  • Category breakdowns: Outflows and inflows by operation type

Sheet 5: Config & Thresholds

  • Detection thresholds: All rules with parameters, rationale, source
  • Categorization rules: Operation type definitions and keyword patterns
  • Exclusion criteria: What was filtered out and why
  • Run-specific parameters: p95 debit, weekend threshold, date ranges, transaction counts
  • QA validation report: Expected vs. actual checks for key metrics (with tolerance thresholds)

Business value and deliverables

Business questions answered

  • Are there duplicate payments that suggest processing errors or fraudulent invoicing?
  • Do payment patterns indicate approval limit circumvention schemes?
  • Which vendors show rapid spending growth requiring risk assessment?
  • Are there unusual transaction patterns (round amounts, weekend activity) needing investigation?
  • Do new high-value counterparties have proper due diligence documentation?
  • Does the transaction dataset exhibit natural statistical properties (Benford's Law)?
  • What is the vendor concentration risk profile?
  • How does cash flow trend over the analysis period?

Deliverables

  • Five-sheet Excel workbook with standardized data and forensic findings
  • Priority-based alert list for efficient reviewer triage
  • Transaction-level evidence with exact source references
  • Cash flow and vendor concentration analytics
  • Complete methodology documentation for audit defensibility

Operational constraints

  • Run from run/ with statement files in the same directory
  • Close Excel workbooks before running (Windows file locking)
  • Ensure sufficient disk space for output workbook (~5-10 MB typical)
  • For large datasets (>10,000 transactions), allow several minutes processing time
  • Input filenames should follow convention: BANKNAME_descriptor.xlsx (e.g., CHASE_20230515.xlsx)

Reproducibility checklist

  1. Create and activate virtual environment: python -m venv .venv → source .venv/bin/activate
  2. Install dependencies: pip install -r requirements.txt (pandas, openpyxl, numpy)
  3. Place statement files in run/ directory
  4. Run processor: cd run → python bank_statement_processor.py
  5. Verify outputs: consolidated_statements.xlsx created with five sheets
  6. Review QA validation in Config & Thresholds sheet — zero tolerance violations indicates successful run
  7. Archive inputs and outputs with consistent naming for version control
